# DSP-013: Backend OpenAPI Refactoring

## Status

**Completed** - 2026-02-27

## Overview

Refactored the backend to adopt baseapp-style architectural patterns with full OpenAPI integration, improving type safety, API documentation, and code maintainability.

## Changes Made

### 1. Baseapp-Style Pattern Implementation

#### Layered Architecture

- **`controller.ts`** → **`service.ts`** → **`repository.ts`** layering
- Clear separation of concerns between HTTP handling, business logic, and data access
- Each layer has a single responsibility

#### Type System Restructuring

- **`modal.ts`** (renamed from `model.ts`): Contains route domain types + OpenAPI schemas
- **`doc.ts`**: OpenAPI route registration using `createRoute` + Zod schemas
- Type-safe request/response handling throughout

#### Response Helpers

- **`Res()` helper**: Maps service responses to HTTP responses consistently
- **`registerHandler()` wrapper**: Eliminates type casts in route handlers
- Standardized error response format

### 2. OpenAPI Integration

#### Dependencies Added

- `@hono/zod-openapi`: Type-safe OpenAPI spec generation from Zod schemas
- `@hono/swagger-ui`: Swagger UI middleware for interactive API docs

#### Endpoints

- **`/openapi.json`**: Generated OpenAPI specification endpoint
- **`/docs`**: Swagger UI for interactive API documentation

#### Schema Management

- Centralized error schemas in `app/rest/openapi.ts`
- Request validation via Zod schemas at route level
- Removed redundant controller/service validation (single source of truth)

### 3. HTTP Logger Middleware

#### Implementation

- **File**: `apps/backend/src/app/middleware/logger.ts`
- **Library**: pino-based request/response logging
- Captures request method, URL, status code, response time
- Structured JSON logging for production environments

#### Exclusions

- `/health`: Health check endpoint (noise reduction)
- `/openapi.json`: OpenAPI spec endpoint
- `/docs`: Swagger UI documentation endpoint

### 4. Cleanup

#### Removed Files

- `routes.ts` wrapper files (functionality replaced by `doc.ts` registrars)
- Screenshots REST endpoint (screenshots are MQTT-only now)
- `baseapp` reference folder
- `external2` reference folder

#### Simplified Structure

- Cleaner route registration pattern
- Less boilerplate code
- More maintainable file organization

## Technical Decisions

### ADR-017: Baseapp-Style Backend Pattern

- **Decision**: Adopt controller/service/repository layering with modal.ts/doc.ts separation
- **Rationale**: Better separation of concerns, type safety, and maintainability
- **Impact**: All backend routes follow consistent pattern

### ADR-018: OpenAPI-First API Design

- **Decision**: Generate OpenAPI specs from code using Zod schemas
- **Rationale**: Single source of truth, always up-to-date documentation
- **Impact**: API consumers have interactive docs at `/docs`

### ADR-019: Request Validation at Route Level

- **Decision**: Use Zod schemas for validation at the route/doc.ts level only
- **Rationale**: Eliminates redundant validation in controllers/services
- **Impact**: Validation happens once, types propagate through layers
